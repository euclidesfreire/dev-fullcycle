# Fluxo de agentes — Livro 1

## 1. Curriculum Architect Agent

Status: Aprovado com ressalvas corrigidas.

Decisões tomadas:

- Organizar o Livro 1 em 24 capítulos, iniciando por computação, hardware, sistemas operacionais, concorrência, memória e redes antes de programação e engenharia de software.
- Separar Internet de HTTP para evitar confundir infraestrutura global com protocolo de aplicação.
- Posicionar Git antes de engenharia de software para que versionamento sustente requisitos, revisão, testes e manutenção.
- Encerrar com pensamento de arquiteto e projeto final integrador, conectando fundamentos às trilhas posteriores de Full Stack, IA e Cybersegurança.

Pendências não bloqueantes:

- Na fase de escrita, cada capítulo deverá receber exemplos completos, diagramas e exercícios graduados.
- A ordem poderá ser reavaliada após revisão técnica dos primeiros capítulos.

## 2. Editorial Director Agent

Status: Aprovado com ressalvas corrigidas.

Revisão editorial:

- A estrutura tem porte de livro técnico, não de apostila, porque combina fundamentos de computação, engenharia de software, qualidade, operação, segurança e arquitetura.
- O livro cria uma base comum antes de especializações, preservando progressão conceitual e conexão com projeto final.
- A criação de arquivos de capítulo foi limitada a esqueletos planejados para cumprir o pedido de não escrever capítulos completos nesta etapa.

Correções exigidas e aplicadas:

- Incluir observabilidade, manutenção e pensamento de arquiteto como capítulos próprios.
- Declarar status `planned` nos arquivos, impedindo interpretação como conteúdo aprovado.
- Manter o template editorial obrigatório em todos os capítulos.

## 3. Depth Auditor Agent

Status: Aprovado com ressalvas corrigidas.

Lacunas apontadas:

- Estruturas rasas por tópico único poderiam deixar sistemas operacionais, redes, testes e arquitetura sem profundidade suficiente.
- O plano precisava explicitar funcionamento interno, trade-offs, segurança e performance em cada capítulo, mesmo antes da escrita completa.
- O projeto final integrador precisava aparecer como mecanismo de consolidação, não apenas como apêndice.

Correções aplicadas:

- Cada capítulo recebeu metadados de pré-requisitos, competências, dependências e conexão com o projeto final.
- Cada arquivo contém todas as seções obrigatórias do template, com orientação editorial para escrita profunda futura.
- O capítulo final foi dedicado a consolidar um sistema mínimo observável, seguro, testável, versionado e preparado para as próximas trilhas.
