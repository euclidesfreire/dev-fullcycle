# Mastery Evidence Map — Matriz de Evidências de Domínio

- **Resumo técnico:** usar para verificar síntese conceitual sem cópia literal.
- **Diagrama:** usar quando relações, camadas, fluxos ou fronteiras forem essenciais.
- **Pseudocódigo:** usar para raciocínio algorítmico antes de implementação.
- **Teste:** usar para comprovar comportamento, regressão, contrato ou hipótese.
- **Checklist:** usar para revisão operacional, segurança, qualidade ou prontidão.
- **ADR:** usar quando houver decisão arquitetural com alternativas e consequências.
- **Threat model:** usar em capítulos com risco, autenticação, dados, abuso ou superfície de ataque.
- **Contrato de API:** usar quando a fronteira entre sistemas, serviços ou clientes for central.
- **Modelo de domínio:** usar para requisitos, entidades, regras, invariantes e linguagem ubíqua.
- **Script:** usar para automação, diagnóstico, validação ou experimento reproduzível.
- **Laboratório:** usar para prática guiada, integração ou simulação de cenário real.
- **Relatório de troubleshooting:** usar para diagnosticar sintomas, hipóteses, evidências e correção.
- **Análise de performance:** usar quando custo, latência, memória, throughput ou escala importarem.
- **Plano de observabilidade:** usar quando operação, incidentes, SLI/SLO, logs, métricas ou traces forem necessários.
- **Decisão arquitetural:** usar para escolhas estruturais que afetam evolução, equipe, custo ou risco.

Toda evidência deve ser revisável, vinculada ao objetivo do capítulo e conectada ao projeto final quando aplicável.
